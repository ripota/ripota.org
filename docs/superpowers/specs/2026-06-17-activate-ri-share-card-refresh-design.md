# Activate RI Share Card Refresh Design

Date: 2026-06-17

## Purpose

Keep the `https://ripota.org/activate-ri-2026/` social share image close to
the live event state without adding runtime image generation. The share image
may lag, but the lag should be bounded and the regeneration path should be
repeatable locally and in CI.

## Background

The Activate RI page currently points Open Graph and Twitter metadata at
`/assets/activate-ri-2026-share-card.png`. That file is committed in the repo
and was captured from the rendered event hero. Unlike the main landing page
share card, it is not generated by an existing mise task.

The main landing page task at `mise/tasks/assets/share-card` is a pure
ImageMagick task. It composites committed images and hardcoded text into
`public/assets/ripota-share-card.png`. That approach works for the main landing
page because its card does not depend on a rendered browser page, Leaflet, live
coverage counts, or client-side data fetching.

The Activate RI card should instead be generated from the rendered page because
the share image is meant to reflect the event hero, map, coverage counts, and
current event state.

## Scope

Add a regeneration path for `public/assets/activate-ri-2026-share-card.png`.

In scope:

- A mise file task for regenerating the Activate RI share card.
- A script that computes a semantic fingerprint before rendering.
- A `--force` option for manual cache bypass.
- A committed metadata file recording the latest fingerprint.
- A GitHub Actions workflow that runs every three hours.
- A `workflow_dispatch` input for forced CI regeneration.
- Committing and pushing changed share-card output from the scheduled workflow.

Out of scope:

- Runtime Open Graph image generation.
- Storing the generated image outside the repository.
- Regenerating the main landing page share card.
- Changing the event page layout specifically for social cards, beyond adding
  stable capture selectors if needed.

## Recommended Approach

Use a Playwright-backed mise task and a scheduled GitHub workflow.

The task should collect the data and template inputs that affect the share
card, hash them, and compare the result to the last committed metadata. If the
fingerprint is unchanged, the task exits without launching a browser unless
`--force` is passed. If the fingerprint changed, the task runs the page locally
with a fixed viewport, captures the event hero as a `1200 x 630` PNG, writes the
image, and updates the metadata file.

The GitHub workflow should run every three hours and use the same mise task.
If the generated image or metadata changes, the workflow commits and pushes
those files.

## Inputs

The fingerprint should cover both event data and visual template files.

Data inputs:

- `public/data/activate-ri-2026/parks.json`
- Live production stops from
  `https://ripota.org/api/activate-ri-2026/public/stops` in scheduled CI mode
- Local/public stops fallback from `public/data/activate-ri-2026/stops.json`
  when production stops are unavailable or when running locally without a
  production data URL
- Event metadata from `src/data/activate-ri-2026/event.ts`

Template inputs:

- `src/pages/activate-ri-2026/index.astro`
- `src/components/activate-ri/EventHero.astro`
- `src/components/ReferenceMap.astro`
- `src/styles/global.css`
- `src/lib/reference-map.ts`
- `src/lib/activate-ri/coverage.ts`
- `src/lib/activate-ri/public-stops-client.ts`

The exact file list may be broadened during implementation if the capture
depends on another shared helper.

## Task Interface

Add a mise task named `assets:activate-ri-share-card`.

Expected usage:

```sh
mise run assets:activate-ri-share-card
mise run assets:activate-ri-share-card -- --force
mise run assets:activate-ri-share-card -- --local-stops
```

Expected flags and environment:

- The default stops source is the production public stops API.
- `--local-stops` uses the committed static stops JSON instead.
- `ACTIVATE_RI_SHARE_CARD_STOPS_URL` optionally overrides the remote stops
  source.

The task should invoke a Node script rather than carrying complex logic in
bash. Node is a better fit for hashing, JSON normalization, HTTP fetches,
metadata writing, local server lifecycle management, and Playwright browser
control.

## Metadata

Write a committed metadata file next to the PNG:

`public/assets/activate-ri-2026-share-card.meta.json`

The metadata should include:

- `fingerprint`
- `generatedAt`
- `inputs.event`
- `inputs.parks`
- `inputs.stops`
- `inputs.template`
- the source URL used for stops, when applicable

The task should update this metadata only when it renders a new card or when
`--force` is used.

## Rendering

The script should start or reuse a local site preview, then capture the
Activate RI overview hero at a fixed social-card viewport.

Rendering requirements:

- Use Playwright from the existing project dependency.
- Use a fixed viewport matching the card dimensions: `1200 x 630`.
- Navigate to `/activate-ri-2026/`.
- Wait for the hero, map, and coverage values to render.
- Capture a stable element or viewport region, not an arbitrary full-page
  screenshot.
- Write exactly `1200 x 630` PNG output.
- Fail clearly if the output dimensions are wrong or required selectors never
  become ready.

The implementation should prefer stable selectors over timing sleeps. Short
timeouts are acceptable around map tile settling if needed, but the main
readiness checks should verify DOM state.

## GitHub Workflow

Add a workflow that runs:

- on `schedule` every three hours
- on `workflow_dispatch`

The manual dispatch input should include a boolean `force` option.

The workflow should:

1. Check out the repository.
2. Install Node dependencies.
3. Install Playwright browser dependencies if the runner does not already have
   them.
4. Run `mise run assets/activate-ri-share-card`, passing `--force` when the
   workflow input requests it.
5. Commit and push only when the PNG or metadata file changed.

The scheduled job should avoid commits when the fingerprint is unchanged.

## Error Handling

If production stops cannot be fetched, the task may fall back to the committed
static stops JSON for local runs. In scheduled CI mode, a failed production
fetch should fail the job rather than silently generating a stale production
card from fallback data.

If rendering fails, the task should leave the previous committed PNG in place
and exit non-zero. The scheduled workflow should not commit partial output.

## Testing

Implementation should verify:

- The task exits without rendering when the fingerprint is unchanged.
- `--force` renders even when the fingerprint is unchanged.
- The generated PNG is exactly `1200 x 630`.
- The metadata file records a current fingerprint and input hashes.
- The workflow command path passes the force flag correctly.

Use focused tests for pure fingerprint and argument behavior where practical.
Use one local task run as integration verification.

## Local And CI Data Sources

The default stops source should be the production public stops API for both
local and scheduled CI runs. This matches the task's purpose: refreshing the
public social image. Local static fallback remains available through
`--local-stops` for offline development or debugging.
