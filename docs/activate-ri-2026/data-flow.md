# Activate RI 2026 Data Flow

This document describes the current Activate RI 2026 data flow and runtime
architecture. It is descriptive, not a proposal.

## System Shape

The project is an Astro static site deployed with Cloudflare Workers Static
Assets. The Worker handles API routes, the protected admin page, and edit-link
shell routing. Static pages and generated JSON live in the Astro build output.

Primary runtime bindings:

- `ASSETS`: serves the Astro static build.
- `DB`: Cloudflare D1 database named `ripota-org`.
- `EMAIL`: Cloudflare Email Service binding for transactional email.

Relevant Worker routing is in `src/worker/index.ts`:

- `/api/activate-ri-2026/*` routes to the Activate RI API handler.
- `/activate-ri-2026/admin/` runs through Cloudflare Access before serving the
  static admin page.
- `/activate-ri-2026/edit/<token>/` rewrites to the static edit shell.
- Other public pages and static JSON files are served by `ASSETS`.

## Sources of Truth

D1 is the source of truth for operational event data:

- activators
- private edit-token hashes
- submitted plans
- activation stops
- approval state
- cancellation state
- activity/audit events

Generated JSON files under `public/data/activate-ri-2026/` are build artifacts
used by public pages for stable event and park reference data. They are not
currently updated by normal admin approval or activator edit API requests.

Official Parks on the Air resources remain the source of truth for POTA rules,
reference metadata, accounts, spots, and logs.

Activation stop times are stored in D1 as UTC instant strings:

- `activate_ri_stops.start_at`
- `activate_ri_stops.end_at`

Public and edit APIs may still expose `plannedDate`, `startTime`, and `endTime`
fields for browser forms and display code. Those fields are derived from the
stored UTC instants.

## Static Versus Live Data

The current public site has two public data paths:

| Data path | Backing source | Updated by approval/edit? | Used for |
| --- | --- | --- | --- |
| Static JSON at `/data/activate-ri-2026/event.json` and `/data/activate-ri-2026/parks.json` | Files generated into `public/data/activate-ri-2026/` before deploy | No. These files change only when regenerated and deployed. | Stable event metadata and park reference data. |
| `GET /api/activate-ri-2026/public/stops` | Short-cached live D1 query over approved plans and public stop statuses | Yes. It reflects D1 after approval or approved activator edits. | Public schedule, park coverage, coverage summary, hero stats, and map coverage. |
| Admin and edit APIs | Live D1 queries and mutations | Yes. These are the operational editing paths. | Organizer review, activity log, activator edit links, cancellation, and resend flows. |

The important consequence is that D1 remains authoritative for operational and
event-dynamic public data. Admin approval and approved activator edits update D1
immediately, and JavaScript-enhanced public views fetch live D1 data through the
public stops API after page load.

## Static Public Data

Static event data is generated by:

```bash
mise run activate-ri-2026:publish
```

That task runs `scripts/activate-ri-2026/publish-public-data.mjs`, which writes:

- `public/data/activate-ri-2026/event.json`
- `public/data/activate-ri-2026/parks.json`

The source for each generated file is:

| Generated file | Source data |
| --- | --- |
| `event.json` | `src/data/activate-ri-2026/event.ts` |
| `parks.json` | `src/data/ri-references.json`, projected into the smaller public park shape by the publish script |

This project no longer generates static `schedule.json` or `coverage.json`
files. Public schedule and coverage state comes from
`GET /api/activate-ri-2026/public/stops`.

The admin API endpoint `POST /api/activate-ri-2026/admin/publish` returns the
current public-safe D1 export rows, but it does not write files or deploy the
site by itself.

## Volunteer Submission Flow

1. The visitor submits `/activate-ri-2026/volunteer/`.
2. `POST /api/activate-ri-2026/plans` validates the JSON payload.
3. Turnstile is verified unless disabled for local development.
4. D1 upserts an activator by normalized email.
5. A private edit token is generated and only its SHA-256 hash is stored.
6. A plan is inserted with `status = 'pending'`.
7. Its stops are inserted with `status = 'pending-review'`.
8. A `plan-created` activity event is written.
9. The Worker attempts to email the private edit link.
10. Email success or failure is written to the activity log.

The submission succeeds even if email delivery fails. The email failure is
visible to admins through the activity log.

This path is covered by a real-SQL acceptance test that applies the checked-in
D1 migrations to a temporary SQLite database. The browser path is covered by the
opt-in `mise run e2e:activate-ri` smoke test.

## Admin Approval Flow

1. An admin opens `/activate-ri-2026/admin/`.
2. Cloudflare Access protects the static admin page in production.
3. The admin UI fetches pending plans from
   `GET /api/activate-ri-2026/admin/plans`.
4. Approval posts to
   `POST /api/activate-ri-2026/admin/plans/<plan-id>/approve`.
5. D1 changes the plan from `pending` to `approved`.
6. D1 changes pending-review stops for that plan to `scheduled`.
7. D1 writes a `plan-approved` activity event.

Approval updates D1 immediately. It does not currently regenerate or deploy the
static JSON files.

## Activator Edit Flow

1. The activator opens `/activate-ri-2026/edit/<token>/`.
2. The Worker serves the static edit shell for that token path.
3. The browser fetches editable plans from
   `GET /api/activate-ri-2026/edit/<token>/plans`.
4. The token is hashed and matched against D1; the raw token is not stored.
5. Saving plan-level changes sends
   `PATCH /api/activate-ri-2026/edit/<token>/plans/<plan-id>`.
6. Saving a single approved stop sends
   `PATCH /api/activate-ri-2026/edit/<token>/stops/<stop-id>`.
7. Cancelling a stop sends
   `POST /api/activate-ri-2026/edit/<token>/stops/<stop-id>/cancel`.
8. Cancelling a whole plan sends
   `POST /api/activate-ri-2026/edit/<token>/plans/<plan-id>/cancel`.

Pending-plan edits keep stops in `pending-review`. Approved-plan edits keep
active stops public by setting edited or added stops to `scheduled`, unless a
stop is already `completed`.

High-impact approved-plan changes attempt to notify admins by email. Those
notifications are best-effort and do not roll back accepted edits.

## Public Schedule And Coverage Flow

Public schedule and coverage surfaces use live D1 data through the public stops
API.

Initial render:

- Static pages can render stable event and park reference data from
  `event.json` and `parks.json`.
- No-JavaScript public pages do not get live schedule or coverage state.

Client refresh:

- Schedule, park coverage, coverage summary, EventHero stats, and map coverage
  components fetch
  `GET /api/activate-ri-2026/public/stops` after page load.
- That endpoint queries D1 for approved plans and public stop statuses.
- The response includes only public-safe stop fields.
- The client renders or updates event-dynamic public data when the fetch
  succeeds.

This means JavaScript is required for live public event schedule state.
JavaScript-enhanced public schedule and coverage surfaces can show approved D1
changes without a deploy.

## Public Export Semantics

The public D1 export includes stops only when:

- the plan has `status = 'approved'`
- the stop status is one of:
  - `scheduled`
  - `delayed`
  - `cancelled`
  - `completed`

The public shape deliberately excludes private fields such as email address,
phone number, edit-token hash, organizer notes, audit data, and raw activity
details.

The transformation from D1 export rows to public stop objects lives in
`src/lib/activate-ri/public-export.ts`.

## Caching

The generated static JSON files under `/data/activate-ri-2026/` have an
explicit browser and shared cache policy through `public/_headers`:

```text
Cache-Control: public, max-age=60, must-revalidate
```

The public stops API sends browser/shared cache headers for short-lived
freshness and also uses the Cloudflare Worker Cache API. Worker Cache API
entries are local to each Cloudflare data center, so a response can be cached in
one colo without being present in another. Admin, edit, and mutation API
responses do not intentionally use public caching.

With this setup, approved D1 changes should appear on JavaScript-enhanced public
views after the next client fetch, subject to the short browser, shared cache,
and per-colo Worker Cache API freshness windows. Open pages do not poll;
visitors must reload or navigate.

## Deployment Flow

Production deploys use:

```bash
mise run deploy
```

The deploy task:

1. verifies Wrangler authentication
2. backs up the production D1 database to `tmp/d1-backups/`
3. applies pending remote D1 migrations
4. deploys the top-level `ripota-org` Worker and static asset build

Do not deploy this project with `wrangler deploy --env production`.

Static event and park JSON changes only become part of the deployed static asset
bundle after the JSON files are regenerated and the site is deployed.

## Current Freshness Boundaries

Current behavior by surface:

- Admin queue: live D1.
- Activator edit pages: live D1.
- Public schedule table with JavaScript: live D1 through the public stops API.
- Public park coverage table with JavaScript: live D1 through the public stops
  API.
- Public reference map with JavaScript: live D1 coverage through the public
  stops API.
- Coverage summary and EventHero stats with JavaScript: live D1 through the
  public stops API.
- No-JavaScript public schedule and coverage: no dynamic schedule or coverage
  state.

The important architectural boundary is that D1 is authoritative for operations,
while static JSON is limited to stable event metadata and park reference data.

## Operational Implications

Admin approval and approved activator edits are immediately represented in D1.
For JavaScript-enabled public schedule and coverage surfaces, they become
visible through the live public stops API without waiting for a deploy.
No-JavaScript public pages do not receive live event schedule state.

At the expected event scale, D1 public reads are comfortably within the paid
plan limits, but the current design does mean public JavaScript traffic can
query D1. A future static-first publishing model could materialize public JSON
after public-data mutations and make D1 strictly an operational source of truth.
